feat(task9): offline queue for Todoist writes — idempotent UUID flush to Inbox

Route Todoist quick-add (global Quick Capture -> Inbox) and check-off through the
existing offline queue. The idempotency key is minted ONCE at enqueue and reused
on every retry, so the proxy lifts requestId -> X-Request-Id and Todoist de-dupes:
a reconnect flush never creates duplicates. Additive — no cutover; the legacy
Notion `cap` flush is untouched (matches Task 8 discipline). Reuses the ONE queue
(lsPush/applyLocal/flushPending) — no second queue, no second flush loop.

cc-data.js (pure, exported)
- queueTodoistAdd({title,area,key}) / queueTodoistDone({id,key}) -> the op the
  client lsPushes. Key minted by idempotencyKey if absent, PRESERVED if present.
  Empty/whitespace title -> null; missing id -> null.
- dedupeQueueByKey(pending): order-preserving collapse of ops sharing a key
  (keeps first; keyless op types always pass through) — defensive against a
  double-enqueue double-creating.
- todoistFlushCall(op,todoistProjects) -> {name,args} reusing buildQuickAddArgs/
  buildCompleteArgs with op.key, so replay carries the SAME requestId. Empty
  title / unknown op -> null. Unknown/global area routes to Inbox.

src/app.c.js
- tdQuickAdd/tdCheckOff: on failure, lsPush the new tdAdd/tdDone op carrying the
  already-minted key + noteQueued() instead of discarding the op. Check-off keeps
  its optimistic removal (op is durably queued); success path unchanged.

src/app.a.js
- flushPending: dedupeQueueByKey(pending) at the top; new tdAdd/tdDone branch
  replays via call(TT.quickAdd|TT.complete, todoistFlushCall(...).args) with the
  SAME op.key. Failures stay in the queue (existing failed[] pattern).
